# Database Setup for PlanMatrix

## Overview

PlanMatrix uses SQL Server (MSSQL) with GUID primary keys for all tables. Database setup is managed through Knex.js migrations.

## Prerequisites

- SQL Server 2019 or later (Express, Standard, or Developer Edition)
- Node.js v18+ with npm
- Connection details configured in `.env.local`:
  - DATABASE_HOST
  - DATABASE_PORT
  - DATABASE_NAME
  - DATABASE_USER
  - DATABASE_PASSWORD

## Initial Setup

### 1. Create the Database and User (One-time)

Use the interactive setup script:

```bash
npm run db:initialize
```

This will guide you through creating the database and user account if needed.

### 2. Run Migrations

Once the database exists, apply all migrations:

```bash
npm run db:migrate
```

This will create all tables with the following schema:

- **floor_plans** — Buildings and floors (GUID primary key)
- **markers** — Cubicle markers on floor plans (GUID primary key)
- **users** — Employees and admins (GUID primary key)
- **assignments** — User-to-marker assignments (GUID primary key)
- **assignment_history** — Audit trail of assignment changes (GUID primary key)
- **settings** — System configuration (GUID primary key)
- **import_logs** — Bulk import history (GUID primary key)

### 3. Verify the Setup

Check migration status:

```bash
npm run db:migrate:status
```

You should see all migrations marked as "completed".

## Database Schema Details

### ID Strategy

All primary keys use `UNIQUEIDENTIFIER` (GUID) generated by SQL Server's `NEWID()` function:
- Secure (non-sequential, hard to predict)
- Globally unique
- Prevents ID enumeration attacks

The application generates GUIDs on insert using Node.js `crypto.randomUUID()`.

### Relationships

- Floor plans → Markers (1:many, cascading delete)
- Users → Assignments (1:1 for active assignments, but 1:many in history)
- Markers → Assignments (1:1 for active, 1:many in history)
- All foreign keys cascade on delete

### Indexes

Indexes are created on:
- `floor_plans.building` — For filtering by building
- `markers.floor_plan_id` — For quick marker lookups
- `assignments.user_id` — For finding a user's current assignment
- `assignments.marker_id` — For finding who occupies a marker
- `assignment_history.user_id` — For audit trail queries
- `assignment_history.timestamp` — For time-range queries
- `import_logs.timestamp` — For log browsing

## Resetting the Database

To start fresh (for development/testing):

```bash
npm run db:reset
```

Or to reset AND recreate the database:

```bash
npm run db:reset:create
```

Then run migrations again:

```bash
npm run db:migrate
```

## Environment Configuration

Create `.env.local` with:

```env
DATABASE_HOST=localhost
DATABASE_PORT=1433
DATABASE_NAME=PlanMatrix
DATABASE_USER=your_username
DATABASE_PASSWORD=your_password
NEXTAUTH_SECRET=your-secret-key-here
PDF_STORAGE_PATH=./public/floor-plans
```

The migration system will use these variables to connect to your SQL Server instance.

## Troubleshooting

### "Login failed for user"
- Check DATABASE_USER and DATABASE_PASSWORD in `.env.local`
- Verify the SQL Server instance is running on DATABASE_HOST:DATABASE_PORT

### "Database does not exist"
- Run `npm run db:initialize` to create it
- Then run `npm run db:migrate`

### "Migration table not found"
- This is created automatically on first migration run. Try again.

### "Cannot insert duplicate key"
- UNIQUE constraints are in place. Check for duplicate building/floor combinations or marker numbers.

### "Foreign key constraint violation"
- Cascading deletes are enabled. This shouldn't happen in normal use.

## Migration Management

### View migration status:
```bash
npm run db:migrate:status
```

### Rollback last migration:
```bash
npm run db:migrate:rollback
```

### Creating new migrations:
```bash
knex migrate:make <migration_name>
```

Migrations use Knex.js syntax. See `db/migrations/` for examples.
