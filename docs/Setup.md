# PlanMatrix Development Environment Setup

This guide walks you through setting up PlanMatrix in a new development environment.

## Prerequisites

- **SQL Server 2019 or later** (Express, Standard, or Developer Edition)
- **Node.js v18+** with npm
- **Git** for version control

## Step 1: Clone and Install Dependencies

```bash
git clone https://github.com/gopinathdee-ai/PlanMatrix.git
cd PlanMatrix
npm install
```

## Step 2: Configure Environment Variables

Create a `.env.local` file in the project root with your SQL Server connection details:

```env
# ==============================================================================
# DATABASE CONFIGURATION
# ==============================================================================
DATABASE_HOST=localhost
DATABASE_PORT=1433
DATABASE_NAME=floorplan_db
DATABASE_USER=floorplan_user
DATABASE_PASSWORD=your_password_here

# ==============================================================================
# AUTHENTICATION & SESSION
# ==============================================================================
# Generate a secure secret for NextAuth.js JWT tokens
# PowerShell command: $b = New-Object byte[] 32; (New-Object System.Security.Cryptography.RNGCryptoServiceProvider).GetBytes($b); [Convert]::ToBase64String($b)
# bash/Linux command: openssl rand -base64 32
NEXTAUTH_SECRET=your-generated-secret-here
NEXTAUTH_URL=http://localhost:3000

# ==============================================================================
# FILE STORAGE
# ==============================================================================
# Directory path where PDF floor plans will be stored
# Must exist and application needs read/write permissions
PDF_STORAGE_PATH=./public/pdfs
```

### Generating NEXTAUTH_SECRET

**On Windows (PowerShell):**
```powershell
$b = New-Object byte[] 32; (New-Object System.Security.Cryptography.RNGCryptoServiceProvider).GetBytes($b); [Convert]::ToBase64String($b)
```

**On macOS/Linux (Bash):**
```bash
openssl rand -base64 32
```

Copy the generated value into your `.env.local` file for `NEXTAUTH_SECRET`.

## Step 3: Prepare SQL Server

1. Open **SQL Server Management Studio**
2. Connect to your SQL Server instance using an account with sysadmin privileges (e.g., `sa`)
3. Open the `scripts/init-db.sql` file in SQL Server Management Studio
4. Set your password in the script:
   - Edit line 1: Replace the empty `@AppPassword` variable with your desired password
   - Example: `DECLARE @AppPassword NVARCHAR(128) = 'smartSong22xuuEK';`
5. Execute the script (F5 or click Execute)

This script will:
- Create the `floorplan_db` database
- Create the `floorplan_user` login
- Grant necessary permissions to the user

## Step 4: Run Database Migrations

Apply all migrations to create the schema:

```bash
npm run db:migrate
```

This will create all tables:
- `floor_plans` — Buildings and floors
- `markers` — Cubicle markers on floor plans
- `users` — Employees and admins
- `assignments` — User-to-marker assignments
- `assignment_history` — Audit trail of assignment changes
- `settings` — System configuration
- `import_logs` — Bulk import history

### Verify Migrations

Check migration status:

```bash
npm run db:migrate:status
```

You should see all migrations marked as "completed". The migrations will output the admin account credentials.

## Step 5: Start the Development Server

```bash
npm run dev
```

The application will be available at **http://localhost:3000**

## Step 6: Login

Access the login page at **http://localhost:3000** and login with the admin credentials displayed during the migration output.

## Database Schema Overview

All tables use GUID primary keys (`UNIQUEIDENTIFIER`) generated by SQL Server's `NEWID()` function:
- Secure (non-sequential, hard to predict)
- Globally unique
- Prevents ID enumeration attacks

### Key Relationships

- Floor plans → Markers (1:many, cascading delete)
- Users → Assignments (1:1 for active, 1:many in history)
- Markers → Assignments (1:1 for active, 1:many in history)

### Indexes Created

- `floor_plans.building` — Filter by building
- `markers.floor_plan_id` — Quick marker lookups
- `assignments.user_id` — Find a user's current assignment
- `assignments.marker_id` — Find who occupies a marker
- `assignment_history.user_id` — Audit trail queries
- `assignment_history.timestamp` — Time-range queries
- `import_logs.timestamp` — Log browsing

## Available npm Scripts

```bash
npm run dev                    # Start development server (http://localhost:3000)
npm run build                  # Build for production
npm start                      # Start production server
npm run lint                   # Run ESLint
npm run db:migrate             # Run all pending migrations

npm run db:migrate:rollback    # Rollback last migration
npm run db:initialize          # Interactive database and user creation
npm run db:reset               # Reset migrations (keep schema)

```

## Troubleshooting

### "Login failed for user"
- Verify `DATABASE_USER` and `DATABASE_PASSWORD` in `.env.local`
- Ensure SQL Server is running on `DATABASE_HOST:DATABASE_PORT`
- Check that the login exists in SQL Server and has proper permissions

### "Database does not exist"
- Run `npm run db:initialize` to create it
- Or manually create via SQL Server Management Studio
- Then run `npm run db:migrate`

### "Cannot connect to SQL Server"
- Verify SQL Server is running: Open `services.msc` (Windows) and check the status
- Check TCP/IP is enabled for SQL Server (SQL Server Configuration Manager)
- Verify firewall allows connections to port 1433

### "Migration table not found"
- This is created automatically on first migration run. Try running migrations again.

### "Port 3000 already in use?"
- Run on a different port: `npm run dev -- -p 3001`

### "Mobile view looks broken"
- Clear browser cache and hard refresh (Ctrl+Shift+R or Cmd+Shift+R)
